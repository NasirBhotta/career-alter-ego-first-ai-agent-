What is an agent?
Abstract
An Agent is an AI-driven system that uses a core AI model to reason, plan, and act within its environment to achieve a user-defined objective, combining thinking (brain) and tool-enabled actions (body) across a spectrum of agency from simple processing to multi-agent workflows.

Key Points
An Agent integrates reasoning, planning, and action execution to fulfill tasks.
The Agent comprises two main parts: the Brain (AI model) and the Body (capabilities/tools).
Agency varies along a spectrum from non-impactful outputs to multi-agent, iterative control.
Large Language Models (LLMs) are the most common core models for Agents, often used with tools.
Tools extend the LLM's capabilities, enabling actions beyond text generation (e.g., image generation, emailing).
Actions and Tools are distinct concepts; Actions may involve multiple Tools.
Agents enable practical, real-life applications like personal assistants, customer service chatbots, and dynamic NPCs in games.
The quality and design of Tools significantly influence the Agent’s performance and usefulness.


What an LLM is?

Large Language Models are AI models that excel at understanding and generating human language.
They are trained on vast text data and typically built on Transformer architecture.
Common types of transformers: Encoders, Decoders, and Seq2Seq (Encoder–Decoder).
Most LLMs today are decoder-based with billions of parameters.
Transformer types and examples

Encoders: e.g., BERT. Use: text classification, semantic search, NER.
Decoders: e.g., GPT-4. Use: text generation, chat, code generation.
Seq2Seq: e.g., T5, BART. Use: translation, summarization, paraphrasing.
Notable models mentioned: GPT-4 (OpenAI), Llama 3 (Meta), SmolLM2 (Hugging Face), Gemma (Google), Mistral.
Core objective and tokenization

The primary objective of an LLM is to predict the next token given a sequence of previous tokens.
Tokens are sub-word units rather than whole words; vocabulary sizes can be in the tens of thousands.
Special tokens exist to open/close sequences, indicate the end of a turn or instruction, etc.
Special tokens

Each model has its own set of special tokens (EOS, end of sequence, end of turn, etc.).
Examples: GPT-4 uses <|endoftext|>, Llama 3 uses <|eot_id|>, SmolLM2 uses <|im_end|>, Gemma uses <end_of_turn>.
These tokens structure generation and input prompts.
Autoregressive decoding and attention

LLMs are autoregressive: output from one step becomes input for the next.
They rely on attention mechanisms to weigh the importance of prior tokens when predicting the next one.
Context length defines how many tokens the model can consider at once.
Prompting and training

Prompt design is crucial for guiding the model toward desired outputs.
Training involves pre-training on large text corpora (self-supervised) to learn language structure, followed by fine-tuning for tasks like conversation or tool use.
How to use LLMs

Run locally if you have sufficient hardware, or use cloud/API solutions (e.g., Hugging Face Inference API).
The course primarily demonstrates usage via Hugging Face Hub APIs, with later exploration of local runs.
LLMs in AI Agents

LLMs act as the “brain” of AI Agents, enabling instruction interpretation, context maintenance, planning, and tool usage decisions.
Learning resources

Links to related courses: free NLP course, Hugging Face NLP course.
Hands-on elements include tokenizers in an interactive playground and decoding strategies (e.g., beam search).



What are Messages and Special Tokens (Messages and Special Tokens.txt)?

Chat templates bridge user-agent conversations and the LLM’s prompt formatting.
User interactions with chat interfaces are translated into a single formatted prompt fed to the model; the model doesn’t retain memory across turns in the UI—it processes the concatenated prompt each time.
Chat templates standardize how multi-turn conversations are serialized into prompts, accommodating differences in special tokens across models.
System messages (system prompts) define persistent behavior for the model (e.g., high-level role, available tools, formatting guidelines).
Conversations consist of alternating user and assistant messages; chat templates help preserve context across turns and maintain coherence.
Example formats show how various models format the same conversation into prompts, including:
SmolLM2-style templates using <|im_start|> and <|im_end|> tokens.
Llama 3’s or other models’ different delimiter schemes.
Base models vs. instruct models:
Base models are trained to predict the next token.
Instruct models are fine-tuned for following instructions and engaging in conversations.
Chat templates help standardize prompts so base models can behave like instruct models.
Using chat templates often involves tooling in transformers (e.g., apply_chat_template) to render the correct textual prompt from a structured message list.
Practical guidance includes examples of converting a message history into a model-ready prompt and references to libraries and templates to handle formatting.
Key takeaway: chat templates ensure consistency and proper formatting across models, enabling multi-turn conversations to be interpreted correctly by the LLM.

This section explains how AI agents perform actions—the concrete steps they take to interact with their environment, such as gathering information, using tools/APIs, manipulating interfaces or devices, and communicating with users. It distinguishes between JSON agents (actions specified in structured JSON, including function-calling as a specialized form) and code agents (actions represented as executable code blocks such as Python), highlighting the benefits and tradeoffs of each approach. A central implementation technique is the stop and parse approach, where the agent outputs a clearly structured action, halts generation once the action definition is complete, and allows an external parser to safely and reliably extract parameters for tool execution.

Key Points
Actions are the deliberate operations an AI agent executes to interact with its environment, bridging internal reasoning and real-world effects.
Agents may perform information gathering by searching the web, querying databases, or retrieving documents.
Agents use tool usage actions such as making API calls, running calculations, and executing code.
Agents may directly perform environment interaction, including manipulating digital interfaces or controlling physical devices.
Agents also handle communication by engaging with users in chat or collaborating with other agents.
A JSON agent represents the intended action as machine-readable JSON, typically containing the tool name and parameters.
A function-calling agent is a specialized variant of the JSON approach designed to invoke a designated function with extracted arguments.
The stop and parse approach improves reliability by forcing the LLM to stop generating once the action is fully defined so output remains parseable.
An external parser reads the formatted action, selects the correct tool/function, and extracts required parameters to prevent errors.
A code agent outputs executable code blocks (often Python), offering greater expressiveness through loops/conditionals and supporting modular reuse.
Executing LLM-generated code introduces security risks (e.g., prompt injection or harmful execution), so the document recommends using agent frameworks with built-in safeguards (like smolagents).
The section concludes that actions prepare the agent for the next stage: using observations to capture environmental feedback for reflection and adaptation.
